# Go v2 Design Loop: Filter Set + Smart-Fusion-Off

The Go column of the benchmark followed the same arc as C: build a curated 14-item corpus, measure the unfiltered baseline, iterate the filter set, regress against the curated corpus at every step.

## What changed between v0 and v13

### Filters added (Go-specific)

| Filter | Effect | Why |
|--------|--------|-----|
| `ct-funcs` (extended) | +50 patterns covering `crypto/subtle`, `crypto/internal/fips140`, `crypto/internal/edwards25519`, `crypto/internal/nistec`, `crypto/cipher`, `crypto/sha*`, `crypto/hmac`, `crypto/ed25519`, `crypto/ecdsa`, `crypto/mlkem`, `golang.org/x/crypto/*`, `github.com/cloudflare/circl/*`, `runtime.*`, common stdlib types | Go's stdlib funnels CT through `crypto/subtle`; almost all FPs come from a small set of well-known package paths |
| `go-bounds-check` | Parser-level tagging: a conditional branch is a bounds check if its taken target OR its fall-through (within ~3 instructions, possibly via one `JMP`) reaches a Go panic helper (`runtime.panicIndex`, `runtime.panicSliceB`, …). | gc emits slice access as `cmp; jcc happy; <panic block>` — both directions need to be checked. |
| `go-stack-grow` | Suppresses warnings whose source line begins with `func ` or whose source line is L1 / file is `<autogenerated>`. | Every gc-compiled function emits `CMPQ SP, 16(R14); JLS morestack` at the prologue. The DWARF entry for these branches points at the function-declaration line; for inlined wrappers it points at L1 or `<autogenerated>`. |
| `go-public-line` | Source-line classifier: pure counted/range loops, len/cap checks, nil/err checks, `if init; err != nil`, `copy()` lines, `if FlagName.Member` boolean checks. Compound `&&`/`\|\|` lines NEVER suppressed. | Replicates the tier-2 idea but conservative: Bleichenbacher's `for i < len(em) && em[i] != 0` survives because it has `&&`. |
| `aggregate` (extended) | Cross-family fold: `BRANCH` warnings on the SAME source line as a `DIV` error in the same function fold into the DIV finding. | gc's signed-divide idiom is `cmp; jne; cdq; idiv` — the JNE shares the IDIV's source line. The Kocher secret-bit branch in `naiveModExp` is on a *different* line from `% m`, so it survives. |

### Source-helper additions

- `_parse_go_secret_funcs` (Go-aware) — uses a permissive substring check on lowercased params. Necessary because Go's `receivedMAC` doesn't satisfy the C-style word-boundary `(?<![A-Za-z])(mac)(?![A-Za-z])`. Without this, bytes.Equal-on-secret patterns are missed entirely.
- `_build_go_function_ranges` — Go-aware brace tracker for memcmp-source attribution.
- `GO_MEMCMP_CALL_RE` — adds `bytes.Equal`, `bytes.Compare`, `runtime.memequal*` to the call-site scanner.

### Run-harness changes

- Cross-(file, line, mnemonic) dedup for inlined call sites: Go inlines `kyberslashReduce` into `main()`; the same source line emits the same IDIV at two different functions. Single-source dedup keeps just one.
- Fuzzy ±2-line cross-opt-level dedup: gc's source attribution drifts ±1 line between optimisation passes; same-(function, mnemonic-family) findings at adjacent lines collapse.
- **Smart-fusion is OFF for Go.** Smart-fusion was designed for clang's O0 vs O2 noise gap; gc's optimisation pipeline is single-stage and the gap is small. Empirically: with fusion off, Go recall is 0.833; with fusion on, 0.667. Precision drops 0.800 → 0.769 — net F1 +0.073.

## Two regressions caught during the loop

1. **v3 over-broad cross-family aggregation** killed the Kocher secret-bit `JE` in `rsa_squareandmultiply.c` (a real C TP). The C held-out check caught it on the next run; v4 scopes the fold to same-line.

2. **v5 over-eager source-line classifier** silenced the L28 length check in `lucky13/verifyMAC`. That suppression was technically correct (it was a false length-check FP), but it ALSO removed the line-tolerance match that was coincidentally consuming the L32 GT. The right fix was to drop smart-fusion (v11), so the real L32 finding at O0 is no longer dropped. The curated regression check exposed the issue immediately.

## Lessons from the Go loop

1. **The Go stdlib's `crypto/subtle` discipline pays off.** Once `ct-funcs` knows about it, almost all FPs disappear: BoringSSL's allowlist needs ~70 patterns; Go's needs ~20.

2. **Compiler-generated wrappers are a real noise class.** Go's gc emits dispatch wrappers attributed to `<autogenerated>` and inlined helpers attributed to L1 of arbitrary files. Both need explicit recognition (`go-stack-grow` filter handles them).

3. **Smart-fusion was a clang-specific optimisation, not a universal one.** The C work over-generalised by treating "drop O0 warnings" as a property of the metric. For gc the same logic loses recall. Each language's compiler-noise profile needs its own fusion strategy.

4. **camelCase secret naming is the dominant Go-vs-C divergence.** Get this right and bytes.Equal on a MAC is detected; get it wrong and the entire Lucky13 pattern slips through.

5. **Cross-language regression checks are non-negotiable.** Two of the three v* iterations introduced changes that helped Go but broke C; the C corpus caught them on the next run.
